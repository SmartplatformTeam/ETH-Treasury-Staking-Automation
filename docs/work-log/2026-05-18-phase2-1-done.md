## 검증된 사실

1. **API 컨테이너 → 호스트 SSH** 정상 동작. 전용 ed25519 key가 ansible role 안에서 idempotent하게 생성/등록되어 컨테이너 내 ansible이 inventory의 `operator_sandbox_1` (ansible_host=host.docker.internal)로 SSH 접속.
2. **cdvn_runtime role의 모든 분기** 동작 확인 — render/verify/rollout(dry-run) 모두 통과. compose execute는 의도적으로 호출 X.
3. **격리 원칙 유지** — 기존 4 obol clone(node0~3 + web3signer + vc)에 어떤 file/process/network 변화도 발생 X.
4. **render output 산출** 확인: `/tmp/cdvn-operator-runtime-team-sandbox/`에 `.env`, `docker-compose.override.yml`, `compose-{el,cl,vc,mev,debug}.yml`, `render-metadata.env` (BASELINE_REF=v1.9.5, OVERLAY_PROFILES=web3signer,observability), `.charon/`, `data/` 생성.
5. **rollout dry-run rsync** 통과 — approval file의 CLUSTER_NAME/HOST_NAME/POLICY mismatch 검증 통과 후 rsync -ain 실행.

## 잠긴 인터페이스 (Phase 2-2+ 재사용)

- 모든 CDVN runtime op는 inventory 그룹 `operator_hosts` 기준 — 베어메탈로 갈 때는 inventory의 `ansible_host`만 실 서버 IP로 교체하면 동일 흐름.
- sandbox와 production cluster를 inventory + secure config 디렉토리만으로 분리.
- approval file 기반의 cluster/host/policy 매칭은 `cdvn_runtime` role의 `assert` task가 강제.

## 알려진 후속 정리 (P2.1 종료 후 별도)

- ansible.builtin.copy가 macOS xattr `._*` 파일까지 호스트로 복사함 — `exclude` 패턴 추가 또는 source dir 정리.
- 현재 sandbox host_name이 inventory(`operator_sandbox_1`)와 host.yml(`operator-sandbox-1`)에 양쪽 명시 — 단일 source 검토.
- rollout `dryRun=true`만 검증함. `dryRun=false`로 execute는 운영 단계까지 보류 (compose execute 안 함).
- macOS xattr 잡파일 제거 후 verify-baseline.sh의 baseline mirror 검증 영향 재확인.
