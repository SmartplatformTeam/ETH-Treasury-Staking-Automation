## 잠긴 결정 (조사 기반 추가)

5. **NodeHealthSnapshot 컬럼**:
   - `id`, `nodeId`, `node` relation (Cascade delete)
   - `beaconHealth: HealthStatus` (HEALTHY/DEGRADED/CRITICAL/UNKNOWN)
   - `isSyncing: Boolean?`
   - `isOptimistic: Boolean?`
   - `isElOffline: Boolean?`
   - `headSlot: BigInt?`
   - `syncDistance: BigInt?`
   - `capturedAt: DateTime` (default now)
   - `@@index([nodeId, capturedAt])`
6. **Node 갱신**:
   - `healthStatus` 갱신 (Beacon health 매핑)
   - `lastHeartbeatAt = now` (성공 시)
   - 신규 컬럼 추가 안 함 — snapshot 에서 시계열 확인
7. **OperatorHost 갱신**:
   - 해당 host 의 모든 Node 중 가장 심각한 healthStatus 로 derive (worst-of)
   - `HEALTHY < DEGRADED < CRITICAL < UNKNOWN`
   - host 단위 다른 source(disk/mem)는 별도 phase
8. **폴링 대상 Node 선정**:
   - `clientType == "lighthouse"` OR `role == "BEACON"` 인 Node (둘 중 어느 게 schema 에 있는지 S2 에서 확인)
   - cluster 단위 1개 Beacon endpoint (현재 `BEACON_BASE_URL` 환경변수 1개) — 모든 cluster 가 같은 endpoint 공유
   - sandbox cluster 와 mainnet cluster 의 Beacon Node row 가 둘 다 있으면 둘 다 같은 endpoint 결과로 갱신
9. **Beacon health → HealthStatus 매핑**:
   - 200 (synced) → HEALTHY
   - 206 (syncing) → DEGRADED
   - 503 (not initialized) → CRITICAL
   - timeout/network error → UNKNOWN
   - syncing 응답에서 `sync_distance > 32` 이면 추가로 DEGRADED
10. **Worker 사이클**: 별도 환경변수 `HEALTH_SYNC_INTERVAL_MS` (default 60000 = 1분). Beacon 보다 더 자주 폴링.
11. **/alerts 코드 변경 없음**: insights.service 가 이미 unhealthy Node/OperatorHost/Signer 합성. 데이터만 들어가면 자동. fixture fallback 만 유지.

## 위험과 대응

| 위험 | 대응 |
|---|---|
| Beacon endpoint 가 잠시 down → 매 사이클마다 UNKNOWN 적재 폭주 | snapshot 은 의도된 동작. UI/alert 에서 노이즈 처리는 P2.7. |
| 같은 endpoint 가 모든 cluster Node 에 매핑 → 모든 Node 가 같은 상태 표시 | 의도된 동작 (단일 endpoint). cluster 별 분리는 별도 phase. |
| Transaction 안에서 여러 Node 갱신 → 한 Node 가 실패하면 모두 rollback | per-node try/catch + per-node transaction. 한 Node 실패가 격리. |
| sandbox cluster 에 BEACON role Node 없음 → 폴링 대상 0 | seed 에 추가. mainnet cluster 의 기존 charon node 는 BEACON 아니라 CHARON. lighthouse 전용 Node row 신설. |
| Worker 가 시간이 오래 걸리는 fetch 로 block | Beacon client 의 fetch 에 AbortController + timeout 5s. |
| Schema 변경 시 기존 batch 진행중인 db push 와 충돌 | 변경 소규모 (1 model + 1 relation). 배포 시 자동 db push. |
