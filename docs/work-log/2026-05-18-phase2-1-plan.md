## 미정 결정 — Architecture (S1 조사 후 결정)

ansible이 API 컨테이너 안에서 실행되는데, render.sh가 호스트의 `/opt/obol/...`에 파일을 만들어야 한다. 두 갈래:

- **옵션 A — 컨테이너 → 호스트 SSH**: inventory의 operator host를 팀 서버 IP/127.0.0.1로, 컨테이너 안에서 SSH로 호스트 접속. 베어메탈 운영과 동일 흐름.
- **옵션 B — 호스트 디렉토리 volume mount**: 호스트 경로를 컨테이너에 mount, ansible `connection: local`. 베어메탈로 가면 재설계 필요.

추천은 A지만, S1 조사로 cdvn_runtime role의 정확한 동작과 현재 컨테이너의 mount 구조를 본 다음 확정.

```text
P2.1-S1. cdvn_runtime role + render/verify/rollout 스크립트 + 현재 컨테이너 mount 조사
   → 산출: docs/work-log/2026-05-18-p2-1-s1-survey.md

   → SSH 경로 또는 volume mount 결정

   → /opt/obol/team-sandbox-obol-a/ (배포 target)

   → 별도 sandbox inventory (~/.config/.../team-sandbox-hosts.yml 또는 기존 inventory에 추가)

P2.1-S4. RENDER_RUNTIME 호출 통과
   → POST /v1/automation/runs operation=RENDER_RUNTIME
   → verify: SUCCEEDED + /opt/obol/team-sandbox-obol-a/에 docker-compose.yml + .env + render-metadata.env 등 생성 확인

P2.1-S5. VERIFY_RUNTIME 호출 통과
   → render output을 검증
   → verify: SUCCEEDED + render output이 baseline version/overlay/secret 규칙 만족

P2.1-S6. ROLLOUT_DRY_RUN 호출 통과
   → sandbox 경로에 rollout approval file 사전 작성
   → POST /v1/automation/runs operation=ROLLOUT_DRY_RUN dryRun=true
   → verify: SUCCEEDED + rsync dry-run 결과가 예상 변경 파일과 일치 + .charon/ 제외 확인
```
