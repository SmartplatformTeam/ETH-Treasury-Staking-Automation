## Phase 2-3 범위 (한 줄)

`POST /approvals/{id}/approve` 와 `POST /approvals/{id}/reject` 두 endpoint를 만들어 execute tier automation 의 사람 승인 체인을 닫는다. 결정 행위는 `AuditLog`에 기록. UI는 결정 버튼 없이 status 가독성만 강화.

## 잠긴 결정 (사용자 confirm)

1. **엔드포인트 범위**: `POST /approvals/{id}/approve` + `POST /approvals/{id}/reject` 두 개만. `GET /:id` 상세와 `POST /approvals` 생성은 다음 phase.
2. **Reject 사유**: 필수 입력 (DTO에서 trim 후 빈 문자열 거부). schema 컬럼 추가 없이 `AuditLog.diff` JSON에만 보관.
3. **자기 승인 금지 (4-eyes principle)**: `Approval.requestedById === session.userId` 면 403. ADMIN 도 예외 없음.
4. **Web UI 범위**: 결정 버튼 추가 안 함. /approvals 페이지에 status 시각화(badge tone), action column 헤더만 정비. 결정은 별도 phase.

## 잠긴 결정 (조사 기반, 추가)

5. **Idempotency**: `Approval.finalStatus` 이미 `APPROVED`/`REJECTED`/`CANCELLED` 이면 409 Conflict. 이중 결정 방지.
6. **상태 전이 허용**: `REQUESTED` 또는 `IN_REVIEW` 만 결정 입력 가능. 그 외는 409.
7. **다단계 미사용**: `Approval.currentStep` 은 single-step (=1) 만 처리. 결정 시 step 증가/검증 안 함.
8. **권한 게이트**: 클래스 레벨 `@RequirePermissions("approvals:read")` 위에 메서드 레벨 `@RequirePermissions("approvals:decide")` 로 덮어쓰기. NestJS 가드는 메서드 메타데이터 우선.
9. **AuditLog actionType**: `APPROVAL_APPROVED` / `APPROVAL_REJECTED`. `resourceType: "Approval"`, `resourceId: approval.id`.

## AGENTS.md 적용

- **§1 Think Before Coding**: 4가지 미정 결정 표면화 → 사용자 confirm → 잠금. 4-eyes/idempotency 같은 보안 정책은 코드보다 먼저 정의.
- **§2 Simplicity First**: schema 변경 없음. AuditLog.diff JSON 으로 reject reason 보관. multi-step은 추후. 새 모듈 만들지 않고 `WorkflowsService`에 메서드 추가.
- **§3 Surgical Changes**: 기존 Approval 모델, ApprovalsController, RBAC 체계, `validateApprovalIfRequired` 모두 그대로. 결정 endpoint 두 개 + Service 메서드 두 개 + UI minor tweak.
- **§4 Goal-Driven**: step별 verify 기준 명시 (curl/swagger + automation execute 시뮬레이션 + audit log 조회).

- `POST /approvals` 생성 endpoint (다음 phase) — execute approval 은 seed/별도 흐름에서 생성
- 다단계 step 진행 (currentStep > 1) — 단일 단계만
- Web UI 결정 버튼/상세 페이지 — 별도 phase

- Multi-tenant approver pool — 현재 stub 6명 user 만 사용
- 실제 ROLLOUT_EXECUTE ansible 실행 — sandbox 외 금지, 시뮬레이션은 dryRun + 가짜 approval

## 격리 원칙 (재확인)

- sandbox cluster `team-sandbox-obol-a` + host `operator-sandbox-1` 만 fixture로 사용
- 기존 obol 운영 컨테이너(node0~3-charon, vc-*, web3signer-*)는 ROLLOUT_EXECUTE에서 호출되더라도 inventory 매칭 안 됨 → automation 단에서 ForbiddenException
- AuditLog는 PII 없음 — actor user id + actionType + diff(JSON) 만
- reject reason 은 사용자 입력 자유 텍스트 → XSS/log injection 방지 위해 길이 cap (예: 1000자) + control char strip

## 위험과 대응

| 위험 | 대응 |
|---|---|
| 자기 승인 우회 (이메일 case 불일치 등) | session.email lookup → user.id 비교. id 단위 매칭. |
| 동시 결정 race condition (두 approver가 동시에 approve/reject) | `prisma.$transaction` + `where: { id, finalStatus: { in: [REQUESTED, IN_REVIEW] } }` 조건부 update. 0 row affected면 409. |
| AuditLog write 실패가 결정 update 를 무효화 못 함 | transaction 안에 두 작업 모두 포함. AuditLog 실패 → 전체 rollback. |
| reject reason 로그 폭주 | 컨트롤러에서 길이 cap, control char strip. |
| 시드 데이터가 다른 환경(prod) 에 새는 위험 | seed 는 sandbox cluster + stub user 만. 실제 운영 approver 는 추후 별도. |
| Approval.finalStatus 가 `IN_REVIEW` 인 fixture 가 P2.3 endpoint 호출로 변하면 다른 테스트가 깨질 수 있음 | seed 가 idempotent — `deleteMany` + `create` 패턴 유지. |

## 잠긴 인터페이스 (P2.4+ 재사용 예정)

- `POST /approvals/{id}/{approve,reject}` 응답: `{ id, finalStatus, approvedById?, rejectedById?, updatedAt }` (read endpoint 응답과 호환)
- AuditLog actionType regex: `^APPROVAL_(APPROVED|REJECTED)$`
- `diff` JSON 스키마: `{ approvedById?, rejectedById?, reason?, previousStatus, policyType, automationOperation? }`
