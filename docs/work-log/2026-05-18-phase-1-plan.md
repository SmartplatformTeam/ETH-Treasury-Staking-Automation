# 2026-05-18 Phase 1 Plan — Control Plane ↔ Ansible end-to-end dry-run

## 범위 (한 줄)

## 잠긴 결정 (사용자 confirm 완료)

1. **Runner 위치**: API 컨테이너 내부에서 `ansible-playbook` child process 실행.
2. **팀 서버 구성**: 1대. Control Plane 컨테이너 + 가짜 operator(localhost / 같은 호스트 별도 디렉토리)가 동거.
3. **첫 playbook**: `infra/ansible/playbooks/verify-baseline.yml` 또는 동등한 read-only 진입점. (없으면 새로 만들거나 기존 role 호출 형태로 생성)
4. **Read 파이프라인 (Phase 3)**: Beacon API 우선.
5. **Sandbox cluster name**: `team-sandbox-obol-a`.
6. **자동화 단위**: 1 operator host. Control Plane은 host별로 automation run을 N번 호출하는 방식 (4대 bundle 흐름은 baseline 점검 외에는 사용 안 함).

## 팀 서버 운영 현황 (격리 대상)

기존에 다음이 떠 있음 — 자동화는 절대 건드리지 않음.

- upstream `ObolNetwork/charon-distributed-validator-node` 레포를 4번 clone해서 charon × 4 운영
- 각 clone에서 `.env`를 수정해 charon + VC 분리 가동
- EL × 1 + CL × 1 공용 (팀 서버 I/O 제약 때문). 나머지 3 operator clone은 공용 EL/CL endpoint를 .env로 바라봄
- DKG는 Obol dashboard에서 진행한 상태

격리 키:

- 자동화 작업 경로: `/opt/obol/team-sandbox-obol-a/`, `/var/lib/eth-treasury-operator-artifacts/team-sandbox-obol-a/`
- Compose project name: `team-sandbox-obol-a` (디렉토리명에서 자동 도출되어 기존 clone 컨테이너와 이름/네트워크 충돌 없음)
- 포트: sandbox는 grafana/prometheus 포트를 기존 운영과 다르게 잡음 (host.yml에서 명시). Phase 2 진입 전 확정.
- 새 EL/CL 절대 안 띄움 (팀 서버 I/O 보호). 필요 시 sandbox host.yml에서 EL/CL profile 제외하거나 dry-run에서 멈춤.

## 가정

- 팀 서버는 Ubuntu 계열, Docker + Compose 사용 가능.
- 현재 `apps/api`는 NestJS, `/v1/automation/runs` endpoint 존재 (read-only 가정).
- 첫 단계에서는 approval gate를 통과시키지 않는다 — read-only 작업은 `automationPolicy: read-only` 분기로 처리.
- 컨테이너에서 호스트의 ansible inventory(`/secure/config/...`) 접근은 bind mount로 해결.
- I/O 약한 팀 서버지만 verify-baseline은 디스크 read만 발생 → 부하 무시 가능.

## 트레이드오프

- API 컨테이너 내부 runner는 시작 단계에서 가장 단순하지만, 동시 실행/취소/장기실행 제어가 약함. **임계점**: 동시 ansible run이 2개 이상 필요해지는 시점에서 worker/queue로 분리. 지금은 분리하지 않는다.
- localhost-loopback ansible 실행은 진짜 SSH 경로를 검증하지 못함. **임계점**: 베어메탈 이관 전에 SSH 경로(team server → 가상 operator host)를 반드시 한 번 통과시킨다. Phase 2 후반에 inventory만 바꿔 검증.

## 비범위 (이번 단계에서 안 함)

- approval gate 연동 (Phase 4)
- queue 기반 worker 분리 (필요 시점까지 보류)
- SSH 기반 원격 ansible (Phase 2 후반)
- Etherscan/Beacon/Obol read 파이프라인 (Phase 3)
- secret/`.charon` artifact 처리 (Phase 4)

## 위험과 대응

- API 컨테이너 이미지 비대화 → multi-stage build, ansible-core만 설치, collection은 필요 시 추가.
- child_process가 죽으면 run 상태가 RUNNING으로 박힘 → start time + max duration으로 timeout/`STALE` 마킹.
- stdout 폭주 시 DB 부담 → 라인 수 cap + truncate marker, 풀 로그는 파일/오브젝트 스토리지로 미루지만 이번에는 단순 DB text 컬럼 + cap.
- **기존 obol 운영 경로 침범** → 자동화는 `team-sandbox-obol-a` cluster 경로 outside 접근 금지. ansible playbook의 `path` 변수에 sandbox 경로 prefix 검증 단계를 추가 (S2에서 어댑터 수준 화이트리스트).
- **컨테이너/포트 충돌** → sandbox compose project name + 다른 grafana/prometheus 포트. Phase 2 진입 전 host.yml 예시에서 잠금.
- **팀 서버 I/O 과부하** → sandbox에서 새 EL/CL 절대 띄우지 않음. Phase 1은 verify-baseline까지라 compose up 자체 없음.
