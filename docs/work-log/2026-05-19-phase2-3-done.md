# 2026-05-19 Phase 2-3 종료 — Approval Decide API + 사람 승인 체인 잠금

## 종료 조건 충족

`POST /approvals/{id}/approve` 와 `POST /approvals/{id}/reject` 두 endpoint 가 운영. APPROVER/ADMIN 만 호출, requester 자기 자신 결정 불가(4-eyes), 이미 결정된 approval 은 409, reject 사유는 필수. 결정 행위는 AuditLog 에 atomic 으로 기록. 격리 sandbox(`team-sandbox-obol-a`) + 기존 obol 운영 무영향.

| 시나리오 | 결과 |
|---|---|
| REQUESTED + INFRA_OPERATOR ROLLOUT_EXECUTE | ✅ 403 "Approval must be APPROVED" |
| APPROVER /approve | ✅ 200, finalStatus=APPROVED, approvedBy populated, AuditLog +1 |
| APPROVED + INFRA_OPERATOR ROLLOUT_EXECUTE | ✅ run QUEUED (즉시 cancel로 ansible 회피) |
| INFRA_OPERATOR /approve (no `approvals:decide`) | ✅ 403 "missing permissions" |
| APPROVER 가 본인 requested 결정 (4-eyes) | ✅ 403 "Requester cannot decide their own approval" |
| APPROVED + /reject | ✅ 409 "already in terminal state 'APPROVED'" |
| APPROVED + /approve (이중 결정) | ✅ 409 |
| /reject body 누락 | ✅ 400 "reason must be a string" |
| /reject empty reason | ✅ 400 "non-empty" |
| /reject 유효 reason | ✅ 200, REJECTED, AuditLog.diff.reason 보관 |

## 검증된 사실

1. **transactional state transition 정상**: `prisma.$transaction` 안에서 `updateMany` 조건부 update (`finalStatus in [REQUESTED, IN_REVIEW]`) + `auditLog.create` 가 atomic. race condition 시 update count=0 → ConflictException → rollback. AuditLog 누수 없음.
2. **AuditLog.diff 스키마 유지**: `{ previousStatus, nextStatus, policyType, automationOperation, reason? }`. reject 시에만 reason 포함.
3. **권한 매트릭스 정상 작동**: INFRA_OPERATOR/TREASURY_OPERATOR/FINANCE_REVIEWER/AUDITOR 가 decide 호출 시 403 "missing permissions". ADMIN/APPROVER 만 통과.
4. **4-eyes 강제**: requester user.id 와 actor user.id 비교. ADMIN 도 예외 없음.
5. **격리 원칙 유지**: scenario 3 에서 ROLLOUT_EXECUTE 가 QUEUED 까지만 가고 즉시 cancel 됨. 실제 ansible run 미트리거. 기존 obol 운영(node0~3, web3signer, vc-*) 무영향.
6. **Seed idempotent**: ansible 재배포로 `db push + db:seed` 가 매번 실행되어도 sandbox approval 이 중복 생성되지 않음 (`deleteMany` scoped to clusterId/hostId/policyType/automationOperation).

## 사전 버그 발견 (P2.3 검증 중)

**Workflows 컨트롤러 DI 실패** — `approvals`, `audit-logs`, `deposits` 세 컨트롤러가 `constructor(private readonly workflowsService: WorkflowsService) {}` 로 정의되어 있었음. tsup/esbuild 가 `emitDecoratorMetadata` 를 emit 못 해서(swc plugin 미설치) NestJS 의 design:paramtypes 기반 DI 가 실패 → `this.workflowsService` 가 undefined → 500.

**증상**: `GET /approvals`, `GET /audit-logs` 가 항상 500. `GET /deposits` 도 동일(권한 RBAC가 먼저 막아서 노출 안 됨).

**왜 발견 안 됐었나**: Web UI 의 `loadApprovals()`, `loadAuditLogs()` 가 `try/catch` 로 감싸서 fallback fixture 로 자동 전환. 사용자가 UI 에서 fixture 데이터를 진짜 데이터로 착각.

**수정**: 세 컨트롤러 constructor 에 `@Inject(WorkflowsService)` 명시. 동일 패턴이 이미 `InventoryController` 에 있었음 (그쪽은 정상 동작).

**Phase 2-3 와의 관계**: 새 endpoint 도 같은 DI 패턴이라 동일하게 500 났을 것. 검증 도중 발견해서 같이 수정. Phase 2-2 까지의 fallback UI 동작은 변하지 않음.

## 잠긴 인터페이스 (Phase 2-4+ 재사용)

- POST `/v1/approvals/{id}/approve` — body 없음, 응답은 list endpoint 와 동일 shape
- POST `/v1/approvals/{id}/reject` — body `{ reason: string }`, 응답은 list endpoint 와 동일 shape
- AuditLog actionType: `APPROVAL_APPROVED`, `APPROVAL_REJECTED`
- AuditLog diff JSON: `{ previousStatus, nextStatus, policyType, automationOperation, reason? }`
- HTTP 코드: 200 (성공), 400 (reason 검증), 403 (RBAC/4-eyes), 404 (approval 없음), 409 (이미 결정됨)
- workflows 컨트롤러 DI 패턴: `@Inject(WorkflowsService)` 명시 (자동 metadata 의존 X)

## 보류/follow-up

- **결정 버튼 UI**: 다음 phase. `/approvals` 상세 페이지 + 결정 폼.
- **POST /approvals (생성 endpoint)**: 다음 phase. 현재는 seed/직접 DB 입력만 가능. INFRA_OPERATOR 가 execute 전에 approval request 만드는 흐름 필요.
- **GET /approvals/{id} 상세**: 다음 phase. AuditLog join 으로 결정 이력 노출.
- **다단계 승인 (currentStep > 1)**: 현재 single-step 만. ApprovalStep 모델 도입 시 별도 phase.
- **Notification (Slack/email) on decide**: out of scope.
- **Approval cancel endpoint**: REJECTED 와 의미 분리되므로 다음 phase.
- **emitDecoratorMetadata 근본 해결**: `@swc/core` 설치하거나 NestJS CLI 빌드 사용으로 모든 컨트롤러의 자동 DI 복원. 지금은 surgical하게 명시 `@Inject` 만 추가.
- **APPROVAL_DECISION_REQUESTED audit event**: 결정 시도 자체(403/409 포함)를 audit 하려면 별도 작업.
- **Cancelled run 자동 cleanup**: scenario 3 검증에서 만든 CANCEL_REQUESTED run 1건이 DB 에 남음. 다음 deploy 시 seed reset 안 함. 운영에서는 별도 정책.
