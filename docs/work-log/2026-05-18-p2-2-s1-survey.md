## 4. 표준 Beacon API endpoints

- `GET /eth/v1/beacon/states/{state_id}/validators/{validator_id}` — 단건. 응답: `balance`, `status` (active_ongoing/pending_initialized 등 12개), `validator.pubkey`, `validator.effective_balance`, `validator.activation_epoch`, `validator.exit_epoch`, `validator.slashed`, `validator.withdrawal_credentials`
- `GET /eth/v1/beacon/states/{state_id}/validators?id=...&id=...` — batch (효율적)
- `GET /eth/v1/beacon/states/{state_id}/validator_balances` — balance만 (가장 가벼움)
- state_id 후보: `head` (live, 가장 빠름), `finalized` (~13분 지연, 안전), `justified` (중간)
- 인증: 대부분 public endpoint는 unauthenticated, Beaconcha.in 같은 곳은 Bearer token

## Phase 2-2 architecture 결정 권고

| 영역 | 권고 | 이유 |
|---|---|---|
| **Endpoint** | **Self-hosted lighthouse 우선** (`http://host.docker.internal:5052`), 미가용 시 fallback **Ankr public** | 운영 mainnet parity, rate limit 없음, API key 부담 X. self-hosted 노출 확인 후 결정 |
| **호출 빈도** | **5분** (epoch ~6.4분과 비슷한 주기) | 너무 자주 호출은 의미 X, 너무 느리면 알림 lag. 표준 monitoring 주기. |
| **DB 저장** | **Validator 모델에 latest balance/status 컬럼 추가** (denormalize) + **별도 `ValidatorBalanceSnapshot` 시계열 테이블** | UI는 latest 필드 한 join 없이 빠르게 표시, 시계열은 reward/audit용으로 분리 |
| **첫 fixture validator** | **mainnet의 active validator pubkey 1개 등록** (사용자가 알려주거나 random Lido 같은 공개 validator) | placeholder pubkey는 Beacon에서 not_found라 실 검증 불가 |
| **Worker 배포 위치** | **control-plane compose에 worker 서비스 추가** (postgres + api + web + worker) | api/web과 같은 이미지/같은 env. 별도 컨테이너로 격리, scheduler는 worker 안에서 |
