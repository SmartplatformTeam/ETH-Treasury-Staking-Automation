## 잠긴 architecture/규칙 (Phase 2-6+ 재사용)

1. **Approval filter API**: `clusterId/hostId/automationOperation` 3개 query — 다른 도메인(Run 상세, deposit 상세 등) 에서도 동일 매칭 검색 가능.
2. **operations.ts 단일 source**: AutomationOperation enum 의 메타데이터(label/mode/dryRun) 가 server action 과 client form 양쪽에서 import. 불일치 방지.
3. **prefill 인코딩**: `base64(JSON{policyType, clusterId, hostId, automationOperation, depositRequestId})`. 다른 페이지에서도 같은 인코딩으로 `/approvals` 진입 가능 (deposit 페이지, run 상세 등).
4. **server action 파일 규칙**: `'use server'` 파일은 async 함수만 export. 상수/타입은 `action-state.ts` 같은 별도 모듈. P2.4 의 hotfix 패턴 일반화.
5. **role-based UI mount**: page 가 session.role 으로 client component 마운트 분기. server-side guard (`assertCreateAllowed`) 는 그대로 (defense in depth).
6. **client-side approval matching**: 페이지 로드 시 APPROVED 전부(limit 100) 가져와 클라이언트에서 filter. 매번 fetch X. 운영 규모 커지면 server-side 전용 endpoint 추가 가능.

## 검증된 사실

1. **dropdown 자동 채움**: APPROVED approval id 가 SSR HTML 에 포함 (정확히 1개 매칭 시 1번 등장 확인).
2. **safe op 일 때 approval 섹션 완전 제거**: 첫 옵션이 safe-tier 라 default selection 으로 page 로드 시 Matching Approval 섹션 0건.
3. **role-based form mount**: AUDITOR/TREASURY_OPERATOR 페이지에 Cluster Select DOM 없음. server-side 렌더 단계에서 제거됨.
4. **prefill 정상 동작**: base64 encoded JSON 디코드 → form expand + select default value 적용. HTML 에 `selected.*value="..."` 패턴 1건 확인.
5. **API filter 보안**: enum 변조와 cuid 형식 변조 모두 400 차단.
6. **server action 깨끗**: web 컨테이너 로그 60초 동안 에러 0건. P2.4 의 'use server' 제약 위반 같은 함정 회피 (action-state 분리 패턴).

## 보류/follow-up

- **브라우저 클릭 submit 검증**: server action 의 실제 POST → run QUEUED 흐름은 curl 로 재현 불편. P2.3 의 server-side gate 검증으로 충분하다고 판단했지만 실제 UI 클릭은 사용자가 확인 권장.
- **client-side filter 한계**: 페이지 로드 시 APPROVED 100건만. 운영에서 데이터 늘면 server-side query 호출로 전환.
- **multiple matching approvals**: 같은 cluster/host/op 에 여러 APPROVED 있을 때 dropdown 에 모두 노출. 자동 선택 정책 없음 (가장 첫 옵션 default). 의도된 동작.
- **prefill 변조**: base64 JSON 안에 미지의 필드 와도 무시. 실제 제출은 server-side `parseCreateApprovalDto` 가 다시 검증. 보안적으로 안전.
- **Run 상세에 approval link back**: 어떤 approval 로 실행된 run 인지 detail 페이지에 link 표시 — follow-up.
- **DepositRequest prefill**: 현재 prefill 디코드 logic 은 depositRequestId 도 지원하지만 트리거할 곳이 없음. 미래 Deposit 페이지에서 사용.
- **stale approval dropdown**: 페이지가 SSR 라서 새 approval 이 생기면 새로고침 필요. revalidatePath 가 form 호출 후 트리거되지만 다른 사용자가 만든 건 자동 갱신 안 됨.
