## 잠긴 architecture/규칙 (Phase 2-5+ 재사용)

1. **Server action 패턴**: `'use server'` 함수가 `postApiJson` 호출 → `revalidatePath` → `ApprovalActionState` 반환. `useActionState` 로 UI 측 pending/error 처리. 다른 도메인 mutation 도 동일 패턴.
2. **`ApiError` 클래스**: API 에러 메시지가 throw 까지 보존 → server action 에서 그대로 사용자에게 표시. NestJS exception filter 가 만든 `{ message, statusCode }` shape 의존.
3. **role-based 생성 매트릭스**: `policyCreationRoles[policyType]` Set 으로 controller 단에서 차단. permission 단위 확장 없이 정책 추가 가능.
4. **권한별 UI 분기**: server component 에서 `hasPermission(session, "approvals:decide")` 평가 후 client component 마운트 여부 결정. UI 측 권한 leak 없음.
5. **UI 컴포넌트 토큰**: `core.tsx` 의 `--miro-*` CSS variable 만 사용. 새 색 안 도입. rounded-none + uppercase tracking 유지.
6. **Audit log 필터링**: 서버 endpoint 는 resourceType 필터만 — resourceId 는 클라이언트 filter (server-side 추가는 follow-up).

## 검증된 사실

1. **server action → API 호출 시 세션 헤더 자동 전파**: `getForwardedHeaders` (Next.js `headers()`) 가 `'use server'` context 에서도 동작. APPROVER 헤더로 들어온 요청이 server action 내부 fetch 까지 그대로 전파.
2. **revalidatePath 가 dynamic route 도 갱신**: `/approvals/${id}` 명시 호출로 결정 후 즉시 detail 페이지가 새 상태 반영.
3. **role-based form 옵션 분기 정확**: TREASURY_OPERATOR 에게는 ROLLOUT 옵션 자체가 form 에 노출 안 됨 (allowedPolicyTypes 가 [DEPOSIT_REQUEST] 한 종류만).
4. **격리 원칙 유지**: sandbox cluster + sandbox host fixture 만 생성 form 의 selector 에 표시. 운영 cluster 도 selector 에 있지만 실제 ROLLOUT_EXECUTE 는 ansible inventory 매칭에서 다시 막힘 (P2.1 잠금).

## 보류/follow-up

- **resourceId 서버측 audit filter**: 현재 `?resourceType=Approval` 만 가능, 클라이언트에서 추가 filter. 데이터 많아지면 API 추가.
- **automation 페이지 통합**: `/automation` 에 "이 작업은 approval 필요" 안내 + approval 드롭다운 자동 채움 → 다음 phase 후보.
- **알림**: 새 approval 생긴 걸 approver 가 알기 위해 매번 새로고침 필요. Slack/email 또는 SSE.
- **Approval cancel endpoint**: REJECTED 와 의미 분리. 작업자가 자기 요청을 철회하는 흐름.
- **POST /approvals 중복 방지**: 같은 cluster/host/operation 으로 진행 중 approval 이 있으면 경고/금지. UI 에서 미리 차단도 가능.
- **나머지 4개 policyType**: DKG_CEREMONY/SAFE_PROPOSAL/SIGNER_BINDING/CHARON_ARTIFACT_STAGE — 해당 도메인 phase 와 함께.
- **DepositDisplayRow.id 빈 문자열 fallback**: fixture fallback 일 때 `id=""` — 진짜 id 없는 경우 UI 가 무링크. 개선 여지.
- **Form expand 시 SSR 미렌더**: 옵션이 client-side에서만 채워짐. 첫 클릭 후 form options 가 보임 — UX 정상이나 SSR 검증으로는 옵션 셀렉터 직접 확인 어려움.
