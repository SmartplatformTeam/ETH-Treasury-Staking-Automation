## Phase 2-4 범위 (한 줄)

운영자가 curl/swagger 없이 Web UI 만으로 approval 을 만들고 결정할 수 있게 한다. P2.3 가 만든 decide API 를 UI 에 노출 + 생성 API 신설.

## 잠긴 결정 (사용자 confirm)

1. **POST /approvals 지원 범위**: `ROLLOUT` + `DEPOSIT_REQUEST` 두 정책만. 다른 4개(`DKG_CEREMONY`, `SAFE_PROPOSAL`, `SIGNER_BINDING`, `CHARON_ARTIFACT_STAGE`) 는 해당 도메인 phase 에서 추가.
2. **UI 구조**: `/approvals` 행 클릭 → `/approvals/[id]` 상세 페이지(결정 버튼 포함), `/approvals` 상단에 inline 생성 form (collapsible).
3. **Submit 방식**: Next.js server action + `revalidatePath('/approvals')`. 클라이언트 상태 최소화, 에러는 `useFormState` 로 인라인 표시.
4. **상세 페이지 audit log**: `?resourceType=Approval&resourceId={id}` 필터로 해당 approval 결정 이력만 하단 노출.

## 잠긴 결정 (조사 기반 추가)

5. **새 UI 컴포넌트**: `packages/ui/src/` 에 `Button`, `Input`, `Textarea`, `FormField` 추가. BMW M 톤(rounded-none, uppercase, hairline border, 흑 캔버스) 유지. Dialog/Modal 은 안 만듦 — reject reason 은 같은 form 안 textarea 로.
6. **권한별 버튼 표시**: `hasPermission(session, "approvals:decide")` 가 true 일 때만 approve/reject 버튼 렌더. `hasPermission(session, "approvals:write")` 같은 새 permission 은 만들지 않음 — 생성은 정책별로 분기:
   - `ROLLOUT` 생성 → `INFRA_OPERATOR`, `ADMIN` 만
   - `DEPOSIT_REQUEST` 생성 → `TREASURY_OPERATOR`, `ADMIN` 만
   - API 컨트롤러에서 `session.role` 직접 검증 (cancelRun 패턴 재사용)
7. **POST /approvals 요청 shape**: `{ policyType, clusterId?, hostId?, automationOperation?, depositRequestId?, resourceType, resourceId }` 중 policyType 별로 필수 필드만 검증. `requestedById` 는 server 가 session 으로 자동 채움.
8. **상세 endpoint shape**: list endpoint 와 동일한 `toApprovalResponse` 재사용 + audit log 별도 query (list 와 같은 shape).
9. **4-eyes 우회 방지**: 생성 시점에 `requestedById === session.user.id` 가 강제됨 → P2.3 decide 의 4-eyes 검증과 자연스럽게 연동.
10. **UI 결정 버튼**: 같은 페이지에 두 form (approve form / reject form) 노출. approve 는 버튼 한 번, reject 는 textarea + 버튼. 별도 dialog 없음 — AGENTS.md "위험 작업은 항상 재확인 단계" 는 결정 페이지 진입 자체가 재확인 단계로 처리.

## AGENTS.md 적용

- **§1 Think Before Coding**: 10개 결정 표면화 → confirm/조사 잠금. 권한 분기, 4-eyes 검증 위치 같은 보안 정책 먼저 정의.
- **§2 Simplicity First**: Dialog/Modal/Toast 컴포넌트 안 만듦. inline form + 에러 메시지. server action 으로 client state 최소화.
- **§3 Surgical Changes**: P2.3 `toApprovalResponse`/`approvalSelect` 헬퍼 재사용. 새 permission 추가 안 함. 기존 audit-logs 컨트롤러 그대로 활용.
- **§4 Goal-Driven**: step 별 verify 기준 명시 (curl + UI 클릭 + audit log 노출 확인).

## 위험과 대응

| 위험 | 대응 |
|---|---|
| server action 안에서 session 헤더 누락 → 401 | `cookies()`/`headers()` 명시 forward, `getForwardedHeaders()` 패턴 확장 |
| 생성 시 같은 resourceId 로 중복 approval | resourceType+resourceId 조합 unique 강제 안 함 — 사용자가 의도적으로 재생성할 수 있음. UI 에서 "최근 동일 자원에 진행 중 approval 있음" 경고 정도 (P2.4 비범위, follow-up) |
| 권한 없는 사용자가 직접 URL `/approvals/[id]` 접근 | 상세 페이지 자체는 `approvals:read` 만 있어도 표시 (목록과 같은 권한). 결정 버튼만 권한 분기 |
| FormData parsing 불일치 (server action vs DTO) | server action 안에서 FormData → plain object 변환 후 `parseCreateApprovalDto(json)` 재사용 |
| UI 컴포넌트 design tone 불일치 (BMW M 톤 깨짐) | core.tsx 의 기존 토큰(`--miro-*`)만 사용. 새 색 도입 금지 |
| Dynamic route 의 server component 가 stale cache 사용 | server action 마다 `revalidatePath` 명시. fetch 옵션은 `cache: 'no-store'` (기존 `fetchApiJson` 정책 확인) |
