## 잠긴 architecture/규칙 (Phase 2-8+ 재사용)

1. **State machine**: `DepositRequest.executionStatus` (DRAFT→EXPORTED→SUBMITTED) + `SafeProposal.status` (DRAFT→EXPORTED→QUEUED→EXECUTED). transition 마다 AuditLog 1건 write in transaction.
2. **Safe Tx Service v1 JSON**: payload shape (safe/to/value/data/operation/safeTxGas/baseGas/gasPrice/gasToken/refundReceiver/nonce/contractTransactionHash/chainId) — 향후 Safe Tx Service 직접 제출 시 그대로.
3. **Domain helper 패턴**: networks/encoding/safe-tx 모두 dependency-free TS. 다른 chain 흐름에서도 재사용.
4. **`loadAuditEntriesForResource` 일반화**: 어떤 resourceType (Approval/DepositRequest/AutomationRun) 도 동일 helper로 표시.
5. **Auto-approval on create**: TREASURY_OPERATOR 가 만든 DepositRequest 는 자동으로 DEPOSIT_REQUEST approval REQUESTED 동반. 별도 호출 없음.
6. **Hex DTO validation**: pubkey(48)/wc(32)/sig(96)/root(32) 길이 + hex 정규식. BLS signature 실 검증은 별도 phase.

## 검증된 사실

1. **encode 결과 정확**: chainId=560048 (Hoodi), to=`0x4242...4242`, value=32 ETH wei (`32000000000000000000`), data=`0x22895118...` + ABI-encoded args (총 714 hex chars).
2. **per-step audit**: 한 deposit 의 lifecycle 에서 `DEPOSIT_REQUEST_CREATED` / `SAFE_PAYLOAD_EXPORTED` / `SAFE_PROPOSAL_SUBMITTED` 3 actionType 이 시간순으로 쌓임.
3. **권한 분기 정확**: TREASURY_OPERATOR + ADMIN 만 write, AUDITOR 는 read 만. 4-eyes 는 P2.3 가드로 그대로 차단.
4. **network 강제 매칭**: cluster.network ≠ request.network 거부, treasury.network ≠ request.network 거부, treasury.chainId ≠ networkConfig.chainId 거부.
