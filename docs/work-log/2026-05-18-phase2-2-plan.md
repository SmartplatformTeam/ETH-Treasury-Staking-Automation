# 2026-05-18 Phase 2-2 Plan — Beacon API read 파이프라인 (validator 실데이터)

## Phase 2-2 범위 (한 줄)

`apps/worker`가 표준 Beacon API를 주기 호출 → validator balance, status, effective balance, activation epoch 등을 DB에 기록 → `apps/web`의 Validators 화면이 fixture 대신 실데이터를 표시.

## 잠긴 결정 (Phase 1/2-1에서 이어짐)

- Sandbox cluster `team-sandbox-obol-a` + production cluster `mainnet-obol-a` 둘 다 seed에 존재.
- API 컨테이너 안에서 ansible 호출 흐름은 그대로 (Phase 2-2는 worker 영역, ansible 무관).
- 격리 원칙: 기존 obol 운영 컨테이너 영향 X.

## 미정 결정 (S1 조사 후)

1. **Beacon endpoint 출처**
   - A. 외부 public endpoint (beaconcha.in, Infura, Alchemy, Ankr 등) — secret/rate limit 트레이드오프
   - B. 자체 노드 (팀 서버에 떠 있는 `node3-cl-lighthouse` 또는 `lighthouse` 직접) — 운영 데이터, dev sandbox에 적합
   - C. 둘 다 — cluster별로 endpoint 다르게 설정 가능
2. **호출 빈도** — Beacon epoch는 ~6.4분. 너무 자주 호출은 의미 X.
3. **DB 저장 방식** — Validator 모델에 컬럼 추가 vs 별도 ValidatorStateSnapshot 모델 (시계열)
4. **Worker가 어떻게 실행되는지** — 현재 compose template에 worker 서비스 있는지, 없으면 추가 필요
5. **첫 validator 데이터** — seed의 placeholder pubkey 사용 vs 실 validator pubkey 등록

## AGENTS.md 적용

- §1 Think Before Coding: endpoint/빈도/저장 방식 결정 표면화 후 진행
- §2 Simplicity First: scheduler는 단순 setInterval 또는 cron, queue 도입은 빈도 늘 때 결정
- §3 Surgical Changes: 기존 Validator 모델/UI 흐름 재사용, 새 read endpoint 추가 최소화
- §4 Goal-Driven: step별 verify 기준 명시

## Step 분해 (잠정)

```text
P2.2-S1. apps/worker + Validator 모델 + Beacon API spec + UI 표시 흐름 조사
   → 산출: docs/work-log/2026-05-18-p2-2-s1-survey.md

P2.2-S2. Architecture 결정 + Beacon client 구현
   → endpoint 출처 선택, env로 URL 주입, http fetch + 타입 정의

P2.2-S3. Worker job 스케줄
   → 주기적으로 validator pubkey 목록 가져와 Beacon API 호출 → DB update
   → idempotent + 실패 격리 (한 validator 실패가 다른 validator에 영향 X)

P2.2-S4. DB 저장 (Validator 모델 또는 신규 snapshot 모델)
   → balance, effectiveBalance, status, lastSyncedAt 등

P2.2-S5. UI/API가 실데이터 표시
   → /v1/inventory/validators가 새 필드 응답
   → apps/web Validators 페이지가 새 필드 표시

P2.2-S6. End-to-end 검증
   → worker가 실 endpoint 호출 → DB 갱신 → UI에 표시 확인
   → 팀 서버 + 로컬 둘 다
```
