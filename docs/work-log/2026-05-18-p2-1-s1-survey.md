# 2026-05-18 P2-1 S1 Survey — render → verify → rollout dry-run 의 현재 상태

## 1. cdvn_runtime role

- 첫 task: `ansible.builtin.copy`로 `{{ cdvn_source_dir | default(playbook_dir ~ '/../../obol-cdvn') }}/` → operator host의 `{{ cdvn_remote_dir }}` (`/tmp/eth-treasury-cdvn`)로 전체 복사. [infra/ansible/roles/cdvn_runtime/tasks/main.yml:2-6](../../infra/ansible/roles/cdvn_runtime/tasks/main.yml)
- `cdvn_runtime_action` 값에 따라 분기:
  - `render`: `/tmp/eth-treasury-cdvn/scripts/render.sh --cluster-file --host-file --output-dir --force`
  - `verify`: `verify.sh --render-dir [--host-name]` (`verify_deployed` 변수로 deployed/rendered 경로 선택)
  - `rollout`: `rollout.sh --render-dir --approval-file --destination [--execute]` (dry-run은 rsync -ain, execute는 -a)
  - `preflight`: `host-preflight.sh --render-dir --deployment-path --min-disk-gb --local --execute`
  - `sync_only`: 첫 copy만 수행 (artifacts/compose/health 등 다른 role과 결합)
- **delegate_to 없음** → playbook의 `hosts: operator_hosts`를 따라 각 operator host에서 실행.
- 즉 ansible은 control-plane container 안에서 실행되어 **SSH로 operator host에 접속**해 명령 수행하는 모델.

## 2. render.sh / verify.sh / rollout.sh input

`render.sh`
- `--cluster-file <path>`, `--host-file <path>` 또는 `--hosts-file`, `--output-dir <path>`, `--force`, `--overlay-profiles`
- 보통 호출: `--cluster-file /secure/config/cluster.yml --host-file /secure/config/host.yml --output-dir /tmp/cdvn-operator-runtime --force`
- output: `.env`, `docker-compose.override.yml`, `render-metadata.env`, `RENDERED_BY_CONTROL_PLANE.txt`, 빈 `.charon/` `jwt/` `data/` 디렉토리

`verify.sh`
- `--render-dir <path>` 필수, `--host-name` 선택
- 검증: baseline version, overlay 정합, `.env` 값, secret 부재 등. exit 0/1.

`rollout.sh`
- `--render-dir`, `--host-name`, `--approval-file`, `--destination`, `--execute`
- `--destination`은 로컬 경로 또는 `user@host:/path`
- rsync 제외: `data/`, `jwt/jwt.hex`, (`--include-charon-artifacts` 없으면) `.charon/`, `charon-artifacts-staging.env`
- dry-run: rsync `-ain` (inventory). execute: rsync `-a --delete`.

## 3. playbooks

`render-runtime.yml`, `verify-runtime.yml`, `rollout-runtime.yml` 모두:
- `hosts: operator_hosts`
- `roles: [cdvn_runtime]` + `cdvn_runtime_action: render|verify|rollout`
- builder가 `--extra-vars`로 `cluster_name`, `host_name`, `deployment_path`, `secure_config_dir`, `runtime_dir`, `approval_file`, `execute`, `dry_run` 등을 주입

## 4. 현재 컨테이너 ↔ 호스트 연결 상태

- [infra/docker/control-plane.Dockerfile:11-13](../../infra/docker/control-plane.Dockerfile) — apt 라인에 `ansible`은 있으나 **`openssh-client` 없음**
- 현재 떠 있는 control-plane API 컨테이너에 **호스트 mount 없음** (compose.dev.yml + ansible compose template 둘 다 호스트 경로 mount 없음)
- 즉 `verify-baseline` (host-agnostic, inline localhost connection)은 통과했지만, render/verify-runtime/rollout은 SSH 또는 mount 없이 호스트 파일 시스템을 건드릴 수 없음

## Architecture 결정 권고

**옵션 A — 컨테이너 → 호스트 SSH (정공법)**
- Dockerfile에 `openssh-client` 추가
- control-plane 전용 SSH key 한 쌍 생성, 컨테이너에 read-only mount, host의 `~/.ssh/authorized_keys`에 등록
- inventory의 sandbox operator host를 `127.0.0.1` (host network) 또는 `host.docker.internal` (extra_hosts) + ansible_user를 host user(smartplatform)로
- `host_key_checking: False` 또는 known_hosts 사전 등록
- **장점**: 베어메탈 이관 시 inventory만 바꾸면 동일 흐름. cdvn_runtime role 동작과 자연 일치.
- **단점**: SSH key setup 1회 작업 추가.

**옵션 B — Volume mount**
- compose.yml에 `/opt/obol`, `/secure/config`, `/var/lib/eth-treasury-operator-artifacts` mount
- playbook 또는 inventory에서 connection: local override
- **장점**: 빠름.
- **단점**: 베어메탈 이관 시 재설계. cdvn_runtime role의 inventory-aware 흐름과 어긋남.

**권고: 옵션 A**. cdvn_runtime role이 inventory + SSH 모델로 이미 설계되어 있고, 베어메탈 이관까지 같은 인터페이스 유지가 핵심 가치.
