## 잠긴 결정 (조사 기반 추가)

5. **Schema 확장 — SafeProposal**:
   - `chainId Int`
   - `to String?` (deposit contract address per network)
   - `value String?` (wei, BigInt → decimal string)
   - `data String?` (hex calldata, `0x22895118` selector + ABI args)
   - `operation Int @default(0)` (Safe CALL)
   - `safeTxGas String @default("0")`
   - `baseGas String @default("0")`
   - `gasPrice String @default("0")`
   - `gasToken String @default("0x0000000000000000000000000000000000000000")`
   - `refundReceiver String @default("0x0000000000000000000000000000000000000000")`
   - `nonce String?` (Safe wallet nonce, 운영자가 Safe UI 보고 입력)
   - `safeTxHash String?` 유지 (계산 결과)
   - `payloadJson Json?` (Safe Tx Service 호환 전체 JSON, 다운로드 직렬화)
   - **기존 `payloadObjectKey` 유지** (P2.9 S3 도입 시 사용)
6. **Network → deposit contract + chainId**:
   - `MAINNET`: `0x00000000219ab540356cBB839Cbe05303d7705Fa`, chainId=1
   - `HOLESKY`: `0x4242424242424242424242424242424242424242`, chainId=17000
   - `HOODI`: `0x4242424242424242424242424242424242424242`, chainId=560048
   - `packages/domain/src/networks.ts` 또는 `packages/config` 에 상수 보관
7. **Deposit calldata**:
   - function: `deposit(bytes pubkey, bytes withdrawal_credentials, bytes signature, bytes32 deposit_data_root)`
   - selector: `0x22895118` (`keccak256("deposit(bytes,bytes,bytes,bytes32)")[:4]`)
   - ABI encoding 직접 구현 (의존성 없이) — 32-byte word align, dynamic types(bytes)는 offset+length+padded data
8. **EIP-712 safeTxHash 계산**:
   - 직접 구현 (외부 라이브러리 없이). `@noble/hashes` 또는 Node 의 `crypto.createHash('keccak256')`... 단 keccak256 은 SHA-3 의 변종이라 Node stdlib 에 없음. `@noble/hashes` 의 keccak_256 사용 (작은 dependency)
   - 우선 P2.7: hash 계산은 `null` 반환 — Safe UI 가 import 시 자체 재계산. follow-up 에 정확 계산.
9. **deposit_data 입력 방식**:
   - `/deposits` 의 [New Deposit] form: 운영자가 deposit_data.json 의 핵심 필드(pubkey, withdrawal_credentials, signature, deposit_data_root) 를 paste
   - validation: 길이/hex format check 만 (BLS signature 실 검증은 P2.8 후보)
   - 우선 단일 validator 1건만 (`validatorCount=1`). 배치는 follow-up.
10. **권한 매트릭스**:
    - `DepositRequest` create: TREASURY_OPERATOR + ADMIN
    - `SafeProposal` export: TREASURY_OPERATOR + ADMIN
    - `Mark Submitted`: TREASURY_OPERATOR + ADMIN
    - `/deposits/[id]` read: `deposits:read` (현재 매트릭스)
11. **State machine**:
    ```
    DepositRequest.validationStatus:  PENDING → VALID (form 통과 후 자동) | INVALID
    DepositRequest.approvalStatus:    REQUESTED (auto-create approval) → APPROVED/REJECTED
    DepositRequest.executionStatus:   DRAFT → EXPORTED (after [Export]) → SUBMITTED (after [Mark Submitted]) → COMPLETED (수동, validator activation 감지 follow-up) | CANCELLED
    SafeProposal.status:              DRAFT → EXPORTED (생성 즉시) → QUEUED (Safe UI 제출 후 mark) → EXECUTED | CANCELLED
    ```
12. **자동 approval 생성**: TREASURY_OPERATOR 가 `POST /deposit-requests` 호출 시 시스템이 DEPOSIT_REQUEST approval(REQUESTED) 도 같이 생성. 단일 transaction. P2.4 의 `createApproval` 패턴 재사용.
13. **AuditLog actionType 신규**:
    - `DEPOSIT_REQUEST_CREATED`
    - `SAFE_PAYLOAD_EXPORTED` (이미 seed 에 같은 이름 있음)
    - `SAFE_PROPOSAL_SUBMITTED`
14. **외부 dependency**: `@noble/hashes` 만 추가 (keccak256 용). ethers/web3 미사용 — ABI encoding 직접.

## 격리 원칙 (재확인)

- DepositRequest 의 clusterId 는 sandbox / mainnet 둘 다 허용. mainnet 경로는 실제 자금이라 사람이 Safe UI 에서 확인하는 단계 보장.
- AGENTS.md 절대 원칙 — validator key seed/mnemonic 절대 DB 저장 X. deposit_data.json 은 pubkey + signature 만이라 sensitive 아님.
- 운영자가 입력하는 hex 값은 길이/형식 검증만 — BLS 검증은 외부 (등록 전 ceremony 결과 검증).
- payload JSON 에는 chainId 포함 — 운영자가 Safe UI 에서 import 시 chain mismatch 발견 가능.

## 위험과 대응

| 위험 | 대응 |
|---|---|
| pubkey/sig hex 길이 잘못 → ABI encoding 깨짐 | DTO 에서 정확한 길이 검증 (pubkey 48B, sig 96B, wc 32B, root 32B = 각각 hex 길이 확인) |
| 운영자가 잘못된 chainId 로 mainnet 자금을 testnet 으로 보낼 위험 | 생성 시 cluster.network 와 chainId 강제 매핑. mismatch 면 400. |
| value(32 ETH = 32e18) 가 hex 가 아닌 decimal 로 인코딩 | Safe Tx Service 표준대로 decimal string. JSON 다운로드 시 명시. |
| nonce 가 outdated → Safe UI 에서 거부됨 | 운영자가 Safe UI 보고 manually 입력. mismatch 발견 시 [Cancel] + 재export. |
| safeTxHash null → Safe UI 가 import 후 자체 계산해서 mismatch 발견 | 의도된 동작. Safe UI 가 source of truth. follow-up 에서 정확 계산. |
| payloadJson 변조 → SQL injection / XSS | Prisma 의 Json 타입은 직렬화 안전. UI 표시는 `<pre>` 안에 escaped. |
| Mark Submitted 후 다시 cancel | cancel 은 EXPORTED 까지만 허용. SUBMITTED 후 cancel 은 의미 없음 (Safe 가 source of truth). |
| 4-eyes 우회 — TREASURY_OPERATOR 가 같은 사람 (자기 만든 deposit 자기 approve) | P2.3 의 자기 승인 금지 그대로 적용. APPROVER 만 별도 사람. |
